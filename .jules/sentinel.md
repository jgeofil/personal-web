## 2026-03-25 - Prevent Sensitive Data Exposure in Analytics URLs
**Vulnerability:** The web vitals reporting logic in `src/lib/vitals.js` was reading the raw `location.href` value and directly appending it to the analytics payload sent to Vercel Analytics. This raw `href` could contain sensitive information like query parameters or URL fragment hashes.
**Learning:** Sending unsanitized URLs directly to external analytics platforms exposes Personally Identifiable Information (PII) or sensitive tokens (e.g., reset tokens, auth tokens) implicitly passed by tracking codes or backend services.
**Prevention:** In analytics endpoints, avoid tracking `location.href` directly. Strip query parameters and hashes from the request payload by logging only `location.origin + location.pathname` to ensure strict analytics hygiene.
## 2024-03-25 - XSS Filter Bypass with Control Characters
**Vulnerability:** The custom `sanitizeUrl` function aimed to block `javascript:`, `data:`, and `vbscript:` protocols. However, it only trimmed standard whitespace (`trim()`) before checking `.startsWith()`. This allowed bypasses using non-standard control characters (e.g., `\x01javascript:alert(1)`) or internal spacing (e.g., `java\nscript:`) that browsers still parse as valid protocols.
**Learning:** Browsers are extremely lenient when parsing URLs and protocols, ignoring many control characters and whitespaces. Simple string prefix checks (`startsWith()`) are insufficient for URL sanitization and prone to bypasses.
**Prevention:** Always use the built-in `URL` constructor to parse URLs. It mirrors browser parsing behavior, including the handling of control characters and whitespace. Fallback regex checks should explicitly strip `^[\s\x00-\x1F]+` to catch all control characters, not just standard whitespace.
